# 2.什么是并发与并行

  • **并行：**指两个或多个事件在同一时刻发生（同时发生）。例如：在一台具有多核cpu的计算机上，不同Cpu同时处理不同的进程，不同CPU之间互不影响。

  • **并发：**指两个或多个事件在同一时间段内发生。例如：在同一时刻，cpu只能处理一条指令，但多个进程指令被快速的轮换执行，使得在宏观上具有多个进程同时执行的效果，但在微观上，并不是同时执行的，而是把实践划分为若干个片段，多个进程快速交替执行。

  • 并发的多个任务之间是互相抢占资源的，而并行的多个任务之间不会互相抢占资源。

# 3.什么是进程与线程

  • 进程：
    1. 进程是正在运行的程序的实例，例如：启动一个word程序，就是启动了一个进程
    2. 进程是线程的容器，即一个进程中可以开启多个线程
  • 线程
    1. 线程是进程内部的一个独立执行单元，例如：在word中，在输入文字的同时，word将会对输入的英文单词做拼写检查
    2. 一个进程可以同时并发运行多个线程

# 4.创建线程

# 4.1java中创建线程的四种方式

  1. 集成Thread类

    public class ThreadDemo extends Thread {

    @Override
    public void run() {
        int size = 10;
        for (int i = 0; i < size; i++) {
            System.out.println("ThreadDemo  " + i + " " + System.currentTimeMillis());
        }
    }

    public static void main(String[] args) {
        ThreadDemo threadDemo = new ThreadDemo();
        threadDemo.start();
        IntStream.range(0, 10).mapToObj(i -> "Main " + System.currentTimeMillis()).forEach(System.out::println);
    }
}

  2. 实现Runnable接口

    public class MyRunnable implements Runnable {
    @Override
    public void run() {
        IntStream.range(0, 10).mapToObj(i -> Thread.currentThread().getName() + "--" + System.currentTimeMillis()).forEach(System.out::println);
    }

    public static void main(String[] args) {

        new Thread(new MyRunnable(), "自定义线程").start();
    }
}

  3. 实现Callable接口

    public class MyCallable implements Callable<String> {
    
    @Override
    public String call() throws Exception {
        IntStream.range(0, 10).mapToObj(i -> Thread.currentThread().getName() + "--" + System.currentTimeMillis()).forEach(System.out::println);
        return "MyCallable接口执行完毕";
    }

    public static void main(String[] args) {
        // 创建FutureTask对象，将Callable接口的实现类对象作为参数
        FutureTask<String> task = new FutureTask<>(new MyCallable());
        // 创建Thread对象，将task作为参数传入
        Thread thread = new Thread(task, "MyCallable");
        // 启动线程
        thread.start();
    }
}

    使用Callable接口创建线程时，会使用FutureTask类

  4. 线程池

    线程池类之间的关系

# 4.2 实现接口与继承Thread类比较

  1. 接口更适合多个相同的程序代码的线程去共享同一个资源
  2. 接口可以避免java中的单继承局限性
  3. 接口代码可以被多个线程共享，代码和线程独立
  4. 线程池只能放入Runnable或Callable接口的线程，不能直接放入继承Thread的类

# 4.3 在java中，每次程序运行至少启动2个线程，一个是main线程，一个是垃圾收集线程

# 4.4 Runnable和Callable接口比较

  1. 相同点
    • 两者都是接口
    • 两者都可以用来编写多线程程序
    • 两者都需要调用Thread.start()启动线程
  2. 不同点
    • 实现Callable接口的线程能返回执行结构，而实现Runnable接口的线程不能返回结果
    • Callable接口的call()方法允许抛出异常，而Runnable接口的run()方法不允许抛出异常
    • 实现Callable接口的线程可以调用Future.cancal取消执行，而实现Runnable接口的线程不能
  3. 注意点
    • Callable接口支持返回执行结果，此时需要调用FutureTask.get()方法实现，此方法会阻塞主线程直到获取到结果，当不调用此方法时，主线程不会阻塞。

# 5.线程生命周期

# 5.1 新建

  • new关键字创建了一个线程之后，该线程就处于新建状态
  • JVM为线程分配内存，初始化成员变量值

# 5.2 就绪

  • 当线程对象调用了start()方法之后，该线程就处于就绪状态
  • JVM为线程创建方法栈和程序计数器，等待线程调度器调度

# 5.3 运行

  • 就绪转台的线程获取CPU资源，开始运行run()方法，该线程进入运行状态

# 5.4 阻塞

当发生如下情况时，线程会进入阻塞状态

  • 线程调用sleep()方法，主动放弃所占用的处理器资源
  • 线程调用了一个阻塞式IO方法，在该方法返回值前，该线程被阻塞
  • 线程试图获得一个同步锁（同步监视器），但该同步锁正被其他线程所持有
  • 线程在等待某个通知（notify）
  • 程序调用了线程的suspend()方法将该线程挂起。但这个方法容易导致死锁，所以应该尽量避免使用该方法

# 5.5 死亡

线程会以如下三种方式结束，结束后就处于死亡状态

  • run()方法或call()方法执行结束，线程正常结束
  • 线程抛出一个未捕获的Exception或Error
  • 调用该线程stop()方法来结束该线程，该方法容易导致死锁，不推荐使用

# 6. 线程安全问题

# 6.1 问题分析

​ 线程安全问题都是由全局变量及静态变量引起的。如果每个线程对全局变量、静态变量只读，不写。一般来说，这个变量是线程安全的。如果有多个线程同时执行写操作，一般都需要考虑线程同步，否则的话就可能影响线程安全。 线程安全的根本原因：

  • 多个线程在操作共享的数据
  • 操作共享数据的线程代码有多条
  • 多个线程对共享数据有写操作

示例：

public class TicketRunnable implements Runnable {

    private int tickets = 10;

    @Override
    public void run() {

        while (true){
            if (tickets > 1) {
                try {
                    Thread.sleep(100);
                } catch (InterruptedException e) {
                    e.printStackTrace();
                }

                System.out.println(Thread.currentThread().getName() + "--" + tickets--);
            }
        }
    }
}

public class TicketRunnableDemo {
    public static void main(String[] args) {
        TicketRunnable ticketRunnable = new TicketRunnable();
        new Thread(ticketRunnable, "线程1").start();
        new Thread(ticketRunnable, "线程2").start();
        new Thread(ticketRunnable, "线程3").start();

    }
}

6.2 问题解决——线程同步

​ 要解决线程安全问题，只要在某个线程修改共享资源的时候，其他线程不能修改该资源，等待修改完毕同步之后，才能去抢夺CPU资源，完成对应的操作，保证了数据的同步性，解决了线程不安全的现象。

​ 为了保证每个线程都能正常执行共享资源操作，Java引入了七种线程同步机制。

  1. 同步代码块——synchronized

    保证当前线程执行时，其他线程只能在同步块外等待。

    public class TicketRunnable implements Runnable {

    private int tickets = 10;

    @Override
    public void run() {

        while (true){
            synchronized (this){
                if (tickets > 0) {
                    try {
                        Thread.sleep(100);
                    } catch (InterruptedException e) {
                        e.printStackTrace();
                    }

                    System.out.println(Thread.currentThread().getName() + "--" + tickets--);
                }
            }
        }
    }
}

  2. 同步方法——synchronized

    保证当前线程执行时，其他线程只能在方法外等待。

    同步锁：

    1. 对于非static方法，同步锁就是this
    2. 对于static方法，同步锁就是当前方法所在类的字节码对象（类名.class）
    public class TicketRunnable implements Runnable {

    private int tickets = 100;

    @Override
    public void run() {
        while (true) {
            this.sell();
        }
    }

    private synchronized void sell() {
        if (tickets > 0) {
            try {
                Thread.sleep(0);
            } catch (InterruptedException e) {
                e.printStackTrace();
            }

            System.out.println(Thread.currentThread().getName() + "--" + tickets--);
        }
    }
}

  3. 同步锁——ReentrantLock

    public class TicketRunnable implements Runnable {

    private int tickets = 10;
    /**
     * fair 是否为公平锁，默认为false（独占锁）
     * true：公平锁，多个线程都公平拥有执行权
     * false：独占锁，如果一个线程获得锁，只有当他主动释放锁或线程执行完毕，其他线程才能获得锁
     *
     */
    private Lock lock = new ReentrantLock(false);

    @Override
    public void run() {
        while (true) {
            lock.lock();
            try {
                if (tickets > 0) {
                    try {
                        Thread.sleep(100);
                    } catch (InterruptedException e) {
                        e.printStackTrace();
                    }

                    System.out.println(Thread.currentThread().getName() + "--" + tickets--);
                }
            } catch (Exception e) {
                e.printStackTrace();
            } finally {
                lock.unlock();
            }
        }
    }
}

# 6.2 synchronized与lock的区别

  1. synchronized是java内置关键字，在jvm层面，Lock是一个类
  2. synchronized无法判断是否获取锁的状态，Lock可以判断是否获取到锁
  3. synchronized会自动释放锁（一个线程执行完毕同步代码或者在执行过程中抛出了异常，都会释放锁），Lock需要手工释放锁，否则容易造成死锁
  4. 使用synchronized关键字修饰方法或代码块，如果一个获得锁的线程进入阻塞状态，其他线程会一直等待下去；而Lock锁就不会一直等待，如果尝试获取锁失败，线程可以不用等待就结束了。
  5. synchronized的锁可重入、不可中断、非公平；而Lock锁可重入、可中断、可公平或独占
  6. synchronized锁是个代码量少的同步问题；Lock锁适合代码量大的同步问题

# 7.线程死锁

​ 多线程以及多进程改善了系统资源的利用率并提高了系统的处理能力。然而，并发执行也带来了新的问题——死锁。

​ 所谓死锁就是指多个线程因竞争资源而造成的一种僵局（互相等待），若无外力作用，这些进程都将无法向前推进。

# 7.1 死锁产生的条件

产生死锁有四个必要条件，缺一不可。

  1. 互斥条件

    进程要求对所分配的资源（如打印机）进行排他性操作，即在一段时间内某资源仅为一个进程所占用。此时若有其他进程请求该资源，则请求进程只能等待。

  2. 不可剥夺条件

    进程所获得的资源在未使用完毕之前，不能被其他进程强行夺走，即只能由获得该资源的进程自己来释放（只能主动释放）。

  3. 请求与保持条件

    进程已经保持了至少一个资源，但又提出了新的请求资源，而该资源已经被其他进程占有，此时请求进程被阻塞，但对自己已经获得的资源保持不放。

  4. 循环等待条件

    存在一种进程资源的循环等待链，链中每一个进程已经获得的资源同时被链中下一个进程所请求。

示例：

public class DeadLockRunnable implements Runnable {

    private Integer flag;

    private static Object object1 = new Object();
    private static Object object2 = new Object();

    public DeadLockRunnable(Integer flag) {
        this.flag = flag;
    }

    @Override
    public void run() {

        if (flag == 1){
            synchronized (object1){
                System.out.println(Thread.currentThread().getName() + "--" + "已经获得object1，正在请求object2");
                try {
                    Thread.sleep(1000);
                } catch (InterruptedException e) {
                    e.printStackTrace();
                }

                synchronized (object2){
                    System.out.println(Thread.currentThread().getName() + "--" + "同时获取object1与object2");
                }
            }
        }else {
            synchronized (object2){
                System.out.println(Thread.currentThread().getName() + "--" + "已经获得object2，正在请求object1");
                try {
                    Thread.sleep(1000);
                } catch (InterruptedException e) {
                    e.printStackTrace();
                }

                synchronized (object1){
                    System.out.println(Thread.currentThread().getName() + "--" + "同时获取object1与object2");
                }
            }
        }

    }
}


public class DeadLockRunnableDemo {
    public static void main(String[] args) {
        DeadLockRunnable deadLockRunnable1 = new DeadLockRunnable(1);
        DeadLockRunnable deadLockRunnable2 = new DeadLockRunnable(2);

        new Thread(deadLockRunnable1, "线程一").start();
        new Thread(deadLockRunnable2, "线程二").start();
    }
}

# 7.2 死锁处理

  1. 预防死锁：通过设置某些限制条件，去破坏产生死锁的四个必要条件中的一个或几个，来防止死锁的发生。
  2. 避免死锁：在资源的动态分配过程中，用某种方法去防止系统进入不安全状态，从而避免死锁的发生。
  3. 检测死锁：允许系统在运行过程中发生死锁，但可以设置检测机构及时检测死锁的发生，并采取适当的措施加以清除
  4. 解除死锁：当检测出死锁后，便采取适当措施将进程从死锁状态中解脱出来

# 7.2.1 死锁预防

​ 预防死锁就是设法至少破坏产生死锁的四个必要条件之一，严格的防止死锁的出现。

  1. 破坏“互斥”条件

    互斥条件时无法破坏的。因此，在死锁预防里主要是破坏其他几个必要条件，而不是破坏互斥条件。

  2. 破坏“占有并等待”条件

    破坏“占有并等待”条件，就是在系统中不允许进程在已经获得某种资源的情况下，申请其他资源。即要想出一个办法，阻止进程在持有资源的同时申请其他资源。

    • 方法一：一次性分配资源，即创建进程时，要求它申请所需要的全部资源，系统或满足其所有要求，或什么都不给它。
    • 要求每个进程剔除新的资源申请前，释放它所占有的资源。这样，一个进程在需要资源S时，必须把它占有的资源R释放掉，然后才能提出对S的申请，即时它可能很快又要用到资源R。

  3. 破坏“不可抢占”条件

    破坏“不可抢占资源”就是允许对资源实行抢夺

    • 方法一：如果占有某些资源的一个进程进行进一步资源请求被拒绝，则该进程必须释放它最初占有的资源，如果有必要，可再次申请这些资源和另外的资源。
    • 方法二：如果一个进程请求当前被另一个进程占有的一个资源，则操作系统可以抢占另一个进程，要求它释放资源。只有在任意两个进程的优先级都不相同的条件下方法二才能预防死锁。

  4. 破坏“循环等待”条件

    破坏“循环等待”条件的一种方法，是将系统中的所有资源统一编号，进程在任何时刻提出资源申请，但所有申请必须按照资源的编号（升序）提出。这样做就能保证系统不出现死锁。

# 8. 线程通讯

​ 多个线程并发执行时，在默认情况下CPU是随机切换线程的，有时我们希望CPU按我们的规律执行线程，此时就需要线程之间协调通信

线程通讯方式：

  • Object的wait、notify、notifyAll
  • Condition的await、signal、signalAll
  • CountDownLatch：用于某个线程A等待若干个其他线程执行完毕之后，它才执行
  • CyclicBarrier：一组线程等待至某个状态之后再全部同时执行
  • Semaphore：用于控制对某组资源的访问权限

# 8.1线程休眠唤醒方式

  1. 使用Object的wait、notify
public class OddEvenDemo {

    private int i = 0;

    public synchronized void odd() {
        while (i < 10){
            synchronized (this){
                if (i % 2 == 1) {
                    System.out.println("奇数：" + i);
                    i++;
                    this.notify();
                } else {
                    try {
                        this.wait();
                    } catch (InterruptedException e) {
                        e.printStackTrace();
                    }
                }
            }
        }
    }

    public void even() {
        while (i < 10){
            synchronized (this){
                if (i % 2 == 0) {
                    System.out.println("偶数：" + i);
                    i++;
                    this.notify();
                } else {
                    try {
                        this.wait();
                    } catch (InterruptedException e) {
                        e.printStackTrace();
                    }
                }
            }
        }
    }


    public static void main(String[] args) {
        OddEvenDemo oddEvenDemo = new OddEvenDemo();

        new Thread(() -> oddEvenDemo.odd()).start();
        new Thread(() -> oddEvenDemo.even()).start();

    }
}
  1. 使用Condition类的signal、await
public class OddEvenDemo {

    private int i = 0;
    private Lock lock = new ReentrantLock();
    private Condition condition = lock.newCondition();

    public synchronized void odd() {
        while (i < 10) {
            lock.lock();
            try {
                if (i % 2 == 1) {
                    System.out.println("奇数：" + i);
                    i++;
                    condition.signal();
                } else {
                    try {
                        condition.await();
                    } catch (InterruptedException e) {
                        e.printStackTrace();
                    }
                }
            } finally {
                lock.unlock();
            }
        }
    }

    public void even() {
        while (i < 10) {
            lock.lock();
            try {
                if (i % 2 == 0) {
                    System.out.println("偶数：" + i);
                    i++;
                    condition.signal();
                } else {
                    try {
                        condition.await();
                    } catch (InterruptedException e) {
                        e.printStackTrace();
                    }
                }
            } finally {
                lock.unlock();
            }
        }
    }


    public static void main(String[] args) {
        OddEvenDemo oddEvenDemo = new OddEvenDemo();

        new Thread(() -> oddEvenDemo.odd()).start();
        new Thread(() -> oddEvenDemo.even()).start();

    }
}
  1. Object与Condition的区别
    • Object wait() 必须在synchronized（同步锁）下使用
    • Object wait() 必须要通过 nodify() 方法进行唤醒
    • Condition await() 必须与Lock（互斥锁/共享锁） 配合使用
    • Condition await() 必须通过 signal() 方法进行唤醒

# 9.Java内存模型

​ Java内存模型是每个java程序员必须掌握理解的，这是java的核心基础，对我们编写代码特别是并发编程时有很大帮助。由于Java程序是交由JVM执行的，所以在谈论java内存区域划分的时候，事实上是指JVM内存区域划分。

# 9.1Java程序执行流程

  1. Java源代码文件会被java编译器编译为字节码文件

  2. 由JVM中的类加载器加载各个类的字节码文件

  3. 加载完成后，交由JVM执行引擎执行

    ​ Java内存模型指的是Runtime Data Area（运行时数据区），即程序执行期间用到的数据和相关信息保存区。

# 9.2java内存模型

​ 根据JVM规范，JVM由五部分组成：虚拟机栈、堆、方法区、程序计数器和本地方法栈。

  1. 程序计数器

    • 每个线程对应有一个程序计数器。
    • 各个线程的程序计数器是线程私有的，互不影响，是线程安全的。
    • 程序计数器记录线程正在执行的内存地址，以便被中断线程恢复时再次按照中断时的指令地址继续执行

  2. Java栈（JVM Stack）

    • 每个线程会对应一个Java栈
    • 每个java栈由若干栈帧组成
    • 每个方法对应一个栈帧
    • 栈帧在方法运行时，创建并入栈；方法执行完，该栈帧弹出栈帧中的元素作为该方法返回值，该栈帧被清除
    • 栈顶的栈帧叫活动栈，表示当前执行的方法，才可以被CPU执行
    • 线程请求的栈帧深度大于虚拟机所允许的深度，将抛出StackOverFlowError异常
    • 栈扩展时无法申请到足够的内存，就会抛出OutOfMemoryError异常

  3. 方法区

    • 方法区是Java堆的永久区（PermanentGeneration）
    • 方法区存放了要加载的类的信息（名称、修饰符等）、类中的静态变量、类中定义为final类型的常量、类中的Field信息、类中的方法信息
    • 方法区是被Java线程共享的
    • 方法区要使用的内存超过其允许的大小时，会抛出OutOfMemoryError：PermGen space 的错误信息

  4. 常量池

    • 常量池是方法区的一部分
    • 常量池中存放两类数据：字面量和引用量
    • 字面量：字符串、final变量
    • 引用量：类、接口、方法和字段的名称和描述等
    • 常量池在编译期就被确定，并保存在已经编译的class文件中

  5. 本地方法栈

    • 本地方法栈与java栈发挥的作用相识，区别是：java栈为JVM执行java方法服务，而本地方法栈为JVM执行Native方法服务
    • 本地方法栈也会抛出OutOfMemoryError和StackOverFlowError

# 9.3java内存模型工作示意图

  1. 类加载器将java代码加载到方法区

  2. 执行引擎从方法区找到main方法

  3. 为方法创建栈帧放入方法栈，同时创建该栈帧的程序计数器

  4. 执行引擎请求CPU执行该方法

  5. CPU将方法栈数据加载到工作内存（寄存器和高速缓存），执行方法

  6. CPU执行完毕之后将执行结果从工作内存同步到主内存

    ​ 线程计算的时候，原始的数据来自内存，在计算过程中，有些数据可能被频繁读取，这些数据被存储在寄存器和高速缓存中，当线程计算完后，这些缓存的数据在适当的时候应该写回内存。

    当多个线程同时读写某个内存数据时，就会产生多线程并发问题。

# 10.线程三大特性

# 10.1 原子性

​ 原子性，即一个操作或者多个操作要么全部执行并且执行过程不能被中断，要么不执行。

# 10.2 可见性

​ 可见性是指当多个线程访问同一个变量时，一个线程修改了这个变量的值，其他线程能够立即看到这个被修改的值。显然，对于单线程来说，不存在可见性问题。

# 10.3 有序性

​ 有序性是指程序的执行顺序按照代码的先后顺序。

# 11.3 CAS的ABA问题及解决

ABA问题：

Thread1在当前获取到内存中的某一变量的值是A，然后Thread1进入阻塞状态。Thread2将该变量的值改为B，然后再改为A；Thread1再次获取此变量的值，发现仍然是A，因此Thread1认为变量的值没有改变。在某些场景下，这样是存在错误风险的。

CAS：CompareAndSwap

ABA问题的解决方案：

使用AtomicStampedReference类，AtomicStampedReference的常用方法：

  1. AtomicStampedReference（初始值，时间戳）：构造函数，设置初始值和初始时间戳
  2. getStamp：获取时间戳
  3. getReference：获取预期值
  4. compareAndSet（预期值，更新值，预期时间戳，更新时间戳）：时间CAS时间戳和预期值的比对
public class ThreadAtomicDemo {

    private static AtomicStampedReference<Integer> atomicStampedReference;

    public static void main(String[] args) {

        for (int j = 0; j < 10; j++) {
            atomicStampedReference = new AtomicStampedReference<>(0, 0);

            Thread thread1 = new Thread(() -> {
                for (int i = 0; i < 1000; i++) {


                    int stamp;
                    Integer reference;

                    do {
                        stamp = atomicStampedReference.getStamp();
                        reference = atomicStampedReference.getReference();
                    } while (!atomicStampedReference.compareAndSet(reference, reference + 1, stamp, stamp + 1));
                }
            });

            Thread thread2 = new Thread(() -> {
                for (int i = 0; i < 1000; i++) {

                    int stamp;
                    Integer reference;

                    do {
                        stamp = atomicStampedReference.getStamp();
                        reference = atomicStampedReference.getReference();
                    } while (!atomicStampedReference.compareAndSet(reference, reference + 1, stamp, stamp + 1));
                }
            });

            thread1.start();
            thread2.start();


            try {
                thread1.join();
                thread2.join();
            } catch (InterruptedException e) {
                e.printStackTrace();
            }


            System.out.println("atomicStampedReference.getReference() = " + atomicStampedReference.getReference());
        }
    }
}

  1. 读写锁

    可以同时读，读的时候不能写；不同同时写，写的时候不能读。

    public class ReadWriteLockDemo {

    private Map<String, String> map = new HashMap<>(16);
    // 定义读写锁
    private ReentrantReadWriteLock lock = new ReentrantReadWriteLock();
    // 定义读锁
    private ReentrantReadWriteLock.ReadLock readLock = lock.readLock();
    // 定义写锁
    private ReentrantReadWriteLock.WriteLock writeLock = lock.writeLock();

    public String get(String key) {
        // 在读之前，加锁
        readLock.lock();

        try {
            System.out.println(Thread.currentThread().getName() + "已加读锁, 开始读。。。");
            try {
                Thread.sleep(0);
            } catch (InterruptedException e) {
                e.printStackTrace();
            }

            return map.get(key);
        } catch (Exception e) {
            e.printStackTrace();
            return null;
        } finally {
            System.out.println(Thread.currentThread().getName() + "读操作结束，解锁");
            // 读操作完毕，解锁
            readLock.unlock();
        }
    }

    public void put(String key, String value) {
        // 在写操作之前，加锁
        writeLock.lock();

        try {
            System.out.println(Thread.currentThread().getName() + "已加写锁，开始写。。。");
            try {
                Thread.sleep(0);
            } catch (InterruptedException e) {
                e.printStackTrace();
            }
            map.put(key, value);
        } catch (Exception e) {
            e.printStackTrace();
        } finally {
            System.out.println(Thread.currentThread().getName() + "写操作结束，解锁");
            // 在写操作完毕后，解锁
            writeLock.unlock();
        }

    }


    public static void main(String[] args) {
        ReadWriteLockDemo lockDemo = new ReadWriteLockDemo();
        new Thread(() -> lockDemo.put("key1", "value1"), "write_1").start();

        new Thread(() -> lockDemo.get("key1"), "read_1").start();
        new Thread(() -> lockDemo.get("key1"), "read_2").start();

        new Thread(() -> lockDemo.put("key2", "value2"), "write_2").start();

        new Thread(() -> lockDemo.get("key2"), "read_3").start();

    }
}

# 11.5 Volatile关键字

一个共享变量（类的成员变量、类的静态成员变量）被Volatile修饰之后，Name就具备了两层语义：

  1. 保证了不同线程对这个变量进行操作时的可见性，即一个线程修改了某个变量的值，这个新值对其他线程来说是立即可见的。（不保证原子性）

  2. 禁止进行指令重排序。（保证变量所在的有序性，不保证所有指令的有序性）

    当程序执行到Volatile变量的读操作或者写操作时，在其前面的操作的更改肯定全部已经进行，且结果已经对后面的操作可见，在其后面的操作肯定还没有进行。

    在进行指令优化时，不能将在对Volatile变量访问的语句放在其后面执行，也不能把Volatile变量后面的语句放到前面执行。

使用Volatile关键字需要满足的条件：

  1. 对变量的写操作不依赖于当前值
  2. 该变量没有包含在具有其他变量的不变式中

使用场景：

  1. 状态量标记
  2. 双重检测（单例）

# 13.四大线程池

# 13.1 为什么使用线程池

多线程的缺点：

  1. 处理任务的线程创建和销毁都非常消耗时间并消耗资源
  2. 多线程之间的切换也会非常耗时并消耗资源

  1. corePoolSize：线程池中核心线程数的最大值

  2. maximumPoolSize：线程池中能拥有最多线程数

  3. maximumPoolSize：用于缓存任务的阻塞队列，对于不同的应用场景我们可能会采取不同的排队策略，这就需要不同类型的阻塞队列，在线程池中常用的阻塞队列有以下两种：

    1. SynchronousQueue:此队列中不缓存任何一个任务。向线程池提交任务时，如果没有空闲线程来运行任务，则入队操作会阻塞。当有线程来获取任务时，出对操作会唤醒入队操作的线程。行这个特性来看，SynchronousQueue是一个无界队列，因此当使用SynchronousQueue作为线程池的阻塞队列时，参数maximumSizes没有任何作用。
    2. LinkedBlockingQueue：顾名思义是用链表实现的队列，可以是有界的，也可以是无界的，但在Executors中默认使用无界的。

  4. corePoolSize、maximumPoolSize、maximumPoolSize之间的关系：

    1. 如果没有空闲的线程执行该任务且当前运行的线程数少于corePoolSize，则添加新的线程执行该任务
    2. 如果没有空闲的线程执行该任务且当前的线程数等于corePoolSize，同时阻塞队列未满，则将任务入队，而不是添加新的线程。
    3. 如果没有空闲的线程执行该任务且阻塞队列已满同时线程池中的线程数小于maximumPoolSize，则创建新的线程执行该任务
    4. 如果没有空闲的线程执行该任务且阻塞队列已满同时线程池中的线程数等于maximumPoolSize，则根据构造函数中的handler指定的策略来拒绝新的任务。

  5. keepAliveTime：表示空闲线程的存活时间

  6. unit：表示keepAliveTime的单位

  7. handler：表示当workQueue已满，且线程池中的线程数达到maximumPoolSize时，线程池拒绝添加新任务时采取的策略。一般采取以下四种值：

    策略 描述
    ThreadPoolExecutor.AbortPolicy() 抛出RejectedExecutionExecption异常
    ThreadPoolExecutor.CallerRunsPolicy() 由线程池提交任务的线程来执行该任务
    ThreadPoolExecutor.DiscardOlddestPolicy() 抛弃最旧的任务（最先提交而没有得到执行的任务）
    ThreadPoolExecutor.DiscardPolicy() 抛弃当前的任务

  8. threadFactory：执行创建的工厂

# 13.3 四种常用的线程池

​ 使用Executors工具类可以简化创建线程池。

  1. newCachedThreadPool

    ​ 该方法创建一个可缓存的线程池，如果线程池长度超过处理需要，可灵活回收空闲线程，若无可回收，则新建线程。

    特点：

    1. 工作线程的创建数量几乎没有限制（IntegerMAX_VALUE）
    2. 空闲的工作线程会自动销毁，有新任务会重新创建
    3. 在使用CachedThreadPool时，一定要注意控制任务的数量，否则，由于大量线程同时运行，有可能会造成系统瘫痪。

  2. newFixedThreadPool

    ​ 该方法创建一个指定工作线程数量的线程池。每当提交一个任务就创建一个工作线程，如果工作线程数量达到线程池初始的最大数，则将提交的任务存入队列中。

    优点：具有线程池提高程序效率和节省创建线程时所耗的开销。

    缺点：在线程池空闲时，即线程池中没有可运行任务时，它不会释放工作线程，还会占用一定的系统资源。

  3. newSingleThreadExecutor

    ​ 该方法创建一个单线程化的Executor，即只创建唯一的工作线程来执行任务，它只会用唯一的工作线程来执行任务，保证所有任务按照指定顺序（FIFO，LIFO，优先级）执行。如果这个不线程异常结束，会有另一个取代它，保证顺序执行。

    ​ 单工作线程最大的特点是可以保证顺序地执行各个任务，并且在任意给定的时间不会有多个线程时活动的。

  4. newScheduleThreadPool

    该方法创建一个定长的线程池，而且支持定时的以及周期性的任务执行。
