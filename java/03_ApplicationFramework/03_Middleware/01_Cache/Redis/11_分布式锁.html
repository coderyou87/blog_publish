# 1.2 没有加锁，在并发情况下，数字会出问题，出现超卖现象

解决：

如果加锁，加 Synchronized、还是 ReentrantLock，还是都可以加。

Synchronized 是 Java 中的关键字，ReentrantLock 是 Java 中的一个类。

Synchronized 是不见不散类型，正常情况下，线程会一直等到获取锁为止，这会导致大量的线程积压。

ReentrantLock 是过时不候类型，ReentrantLock 中有一个 tryLock 方法，如果尝试获取失败，可以放弃或再次获取，或者在一定时间内不断尝试获取锁，超时放弃。

@RestController
public class RedisController {

    @Autowired
    private StringRedisTemplate redisTemplate;

    @Value("${server.port}")
    private String serverPort;

    private final ReentrantLock lock = new ReentrantLock();

    @GetMapping("/buyGoods")
    public String buyGoods() {

        synchronized (this) {
            System.out.println("serverPort = " + serverPort);
            ValueOperations<String, String> operations = redisTemplate.opsForValue();
            String result = operations.get("goods:001");
            int goodNumber = result == null ? 0 : Integer.parseInt(result);

            if (goodNumber > 0) {
                int nextNumber = goodNumber - 1;
                //operations.decrement("goods:001");
                operations.set("goods:001", String.valueOf(nextNumber));
                System.out.println("buy goods success");
                System.out.println("nextNumber = " + nextNumber);

                return "购物成功，当前库存余量为：" + nextNumber;
            }
            return "购物失败，库存余量为0";
        }
    }
}

# 1.3 分布式部署后，单机锁还是会出现超卖现象，需要分布式锁

使用 Nginx 做负载均衡，在后台启动两个微服务，使用 Jemeter 压测，就能够看到超卖的情况

Nginx 配置：

worker_processes  1;

events {
    worker_connections  1024;
}

http {
    include       mime.types;
    default_type  application/octet-stream;

    sendfile        on;
    keepalive_timeout  65;

    upstream myNginx {
        server 192.168.0.104:1111 weight=1;
        server 192.168.0.104:2222 weight=1;
    }

    server {
        listen       80;
        server_name  localhost;

        location / {
            proxy_pass http://myNginx;
            index index.html index.htm;
        }

        error_page   500 502 503 504  /50x.html;
        location = /50x.html {
            root   html;
        }
    }
}

解决方法：

使用 Redis 的 setnx 命令

@RestController
public class RedisController {

    @Autowired
    private StringRedisTemplate redisTemplate;

    @Value("${server.port}")
    private String serverPort;

    public static final String REDIS_LOCK = "redisLock";

    @GetMapping("/buyGoods")
    public String buyGoods() {
        String value = UUID.randomUUID().toString() + Thread.currentThread().getName();
        // 加锁 set nx
        Boolean flag = redisTemplate.opsForValue().setIfAbsent(REDIS_LOCK, value);
        if (Boolean.TRUE.equals(flag)) {
            ValueOperations<String, String> operations = redisTemplate.opsForValue();
            String result = operations.get("goods:001");
            int goodNumber = result == null ? 0 : Integer.parseInt(result);
            if (goodNumber > 0) {
                int nextNumber = goodNumber - 1;
                operations.set("goods:001", String.valueOf(nextNumber));
                // 解锁
                redisTemplate.delete(REDIS_LOCK);
                System.out.println("serverPort = " + serverPort + "   nextNumber = " + nextNumber);
                return "购物成功，当前库存余量为：" + nextNumber;
            }
            redisTemplate.delete(REDIS_LOCK);
            return "购物失败，库存余量为0";
        } else {
            //System.out.println("serverPort = " + serverPort + "   加锁失败");
            return "抢锁失败";
        }
    }
}

# 1.5 服务器宕机

// 过期自动释放锁，设置key的过期时间

# 1.7 删除了别人的锁

A 线程加锁后，到期后还没有完成业务处理，redis 自动释放锁，B 线程加锁，在经过一定时间后，A 线程业务处理完毕，执行释放锁的程序，而此时删除的确实 B 线程的锁。

解决方法：在释放锁之前做判断，只删除自己的锁

if (value.equalsIgnoreCase(redisTemplate.opsForValue().get(REDIS_LOCK))) {
	redisTemplate.delete(REDIS_LOCK);
}

# 1.8 在释放锁的时候，不是原子操作

加锁与解锁不是同一个客户端，存在误解锁的情况。

解决方法：使用 lua 脚本

// 使用lua脚本释放锁
try (Jedis jedis = RedisUtils.getJedis()) {
    String script = "if redis.call(\"get\",KEYS[1]) == ARGV[1]\n" +
            "then\n" +
            "    return redis.call(\"del\",KEYS[1])\n" +
            "else\n" +
            "    return 0\n" +
            "end";
    Object eval = jedis.eval(script, Collections.singletonList(REDIS_LOCK), Collections.singletonList(value));
    if ("1".equalsIgnoreCase(String.valueOf(eval))) {
        System.out.println("del ok");
    } else {
        System.out.println("del fail");
    }
}

问题：不使用 lua 脚本，如何实现

解决方法：使用 Redis 事务

  • Redis 的事务是通过 MULTI、EXEC、DISCARD、WATCH 这四个命令来完成
  • Redis 的单个命令都是原子性的，所以这里确保事务性的对象是命令集合
  • Redis 将命令集合序列化，并确保处于一个事物的命令集合连续且不被打断的执行
  • Redis 不支持回滚
// 用事务来决绝原子性问题
while (true) {
    redisTemplate.watch(REDIS_LOCK);
    if (value.equalsIgnoreCase(redisTemplate.opsForValue().get(REDIS_LOCK))) {
        redisTemplate.setEnableTransactionSupport(true);
        redisTemplate.multi();
        redisTemplate.delete(REDIS_LOCK);
        List<Object> list = redisTemplate.exec();
        if (list.isEmpty()) {
            continue;
        }
    }
    redisTemplate.unwatch();
    break;
}

# 1.9 确保 redisLock 的过期时间大于业务时间（Redis 分布式锁如何续期）

Redisson 中存在看门狗机制，能够自动续期

redis：AP

zookeeper：CP

redisson 的使用：

依赖：

<dependency>
    <groupId>org.redisson</groupId>
    <artifactId>redisson</artifactId>
    <version>3.13.6</version>
</dependency>

配置：

@Bean
public Redisson redisson() {
    Config config = new Config();
    SingleServerConfig singleServerConfig = config.useSingleServer();
    singleServerConfig.setAddress("redis://192.168.76.6:6380");
    singleServerConfig.setPassword("coder");
    singleServerConfig.setDatabase(0);
    return (Redisson) Redisson.create(config);
}

业务：

@Autowired
private Redisson redisson;
RLock redissonLock = redisson.getLock(REDIS_LOCK);
redissonLock.unlock();

@RestController
public class RedisController {

    @Autowired
    private StringRedisTemplate redisTemplate;

    @Value("${server.port}")
    private String serverPort;

    @Autowired
    private Redisson redisson;

    public static final String REDIS_LOCK = "redisLock";

    @GetMapping("/buyGoods")
    public String buyGoods() {
        RLock redissonLock = redisson.getLock(REDIS_LOCK);
        redissonLock.lock();
        try {
            ValueOperations<String, String> operations = redisTemplate.opsForValue();
            String result = operations.get("goods:001");
            int goodNumber = result == null ? 0 : Integer.parseInt(result);
            if (goodNumber > 0) {
                int nextNumber = goodNumber - 1;
                operations.set("goods:001", String.valueOf(nextNumber));
                System.out.println("serverPort = " + serverPort + "   nextNumber = " + nextNumber);
                return "购物成功，当前库存余量为：" + nextNumber;
            }
            return "购物失败，库存余量为0";
        } finally {
            redissonLock.unlock();
        }
    }
}

以上代码在超高并发情况下，可能会出现的问题：

解锁代码优化：

if (redissonLock.isLocked()) {
    if (redissonLock.isHeldByCurrentThread()) {
        redissonLock.unlock();
    }
}

# Redisson

Redisson (opens new window)是架设在Redis (opens new window)基础上的一个 Java 驻内存数据网格（In-Memory Data Grid）。充分的利用了 Redis 键值数据库提供的一系列优势，基于 Java 实用工具包中常用接口，为使用者提供了一系列具有分布式特性的常用工具类。使得原本作为协调单机多线程并发程序的工具包获得了协调分布式多机多线程并发系统的能力，大大降低了设计和研发大规模分布式系统的难度。同时结合各富特色的分布式服务，更进一步简化了分布式环境中程序相互之间的协作。

使用：

<dependency>
    <groupId>org.redisson</groupId>
    <artifactId>redisson</artifactId>
    <version>3.13.6</version>
</dependency>

配置：

@Bean
public RedissonClient redissonClient() {
    Config config = new Config();
    SingleServerConfig singleServerConfig = config.useSingleServer();
    singleServerConfig.setAddress("redis://192.168.76.6:6380");
    singleServerConfig.setPassword("coder");
    singleServerConfig.setDatabase(0);
    return Redisson.create(config);

@RestController
public class RedisController {

    @Autowired
    private StringRedisTemplate redisTemplate;

    @Value("${server.port}")
    private String serverPort;

    @Autowired
    private RedissonClient redissonClient;

    public static final String REDIS_LOCK = "redisLock";

    @GetMapping("/buyGoods")
    public String buyGoods() {
        RLock redissonClientLock = redissonClient.getLock(REDIS_LOCK);
        try {
            redissonClientLock.lock();
            ValueOperations<String, String> operations = redisTemplate.opsForValue();
            String result = operations.get("goods:001");
            int goodNumber = result == null ? 0 : Integer.parseInt(result);
            if (goodNumber > 0) {
                int nextNumber = goodNumber - 1;
                operations.set("goods:001", String.valueOf(nextNumber));
                System.out.println("serverPort = " + serverPort + "   nextNumber = " + nextNumber);
                return "购物成功，当前库存余量为：" + nextNumber;
            }
            return "购物失败，库存余量为0";
        } finally {
            if (redissonClientLock.isLocked()) {
                if (redissonClientLock.isHeldByCurrentThread()) {
                    redissonClientLock.unlock();
                }
            }
        }
    }
}

Redisson 中的锁，默认过期时间时 30 秒，到期自动解锁。

自动续期：使用 lock.lock()方法，进行加锁，能够自动续期。默认续期时间是 30 秒，每隔 10 秒（默认续期时间的三分之一）都会自动续期一次。

如果在加锁的时候，指定了锁的过期时间，则不能自动续期。

# 数据一致性（Redis 与数据库）

  1. 双写模式

    修改完数据库，就立即修改缓存中的数据。

    漏洞：存在ABA问题。会产生脏数据。

    解决方法：给操作数据库和写缓存的时候，加一把锁。如果允许数据短暂不一致，则可以忽略这个问题，因为在写入缓存的时候，给了key一个过期时间，key过期就被删除了，后面查询的时候，就能查询到新的数据。

  2. 失效模式

    修改完数据库，直接删除缓存中的数据。等待下次查询时，更新数据。

    漏洞：任然存在ABA问题

    解决方法：可以加锁。

经常修改的数据，应该直接操作数据库，而不是缓存。

缓存数据一致性解决方案：

  • 无论是双写模式还是失效模式，都会导致缓存数据与数据库不一致的问题。即多个实例同时更新。
    1. 如果是用户纬度数据（订单数据，用户数据），这种并发几率非常小，不用考虑这个问题，缓存数据加上过期时间，每隔一段时间触发读操作，主动更新数据。
    2. 如果是菜单、商品介绍等数据，也可以使用canal订阅binlog的方法
    3. 缓存数据 + 过期时间也足够解决大部分缓存需求
    4. 通过加锁保证并发读写，写写的时候排队操作，读读可同时进行。可以使用读写锁完成。
    5. 如果数据允许暂时不一致，可不用考虑数据不一致问题，使用缓存+过期时间就能满足需求
  • 总结
    1. 放入缓存的数据不应该是实时性、一致性要求非常高的数据，在缓存数据的时候加上过期时间，这样可以保证数据不一致的时间不会太久。
    2. 不应该过度设计，增加系统的复杂性
    3. 遇到实时性、一致性要求非常高的数据，直接查询数据库。
