# Redis 配置文件

# 一、网络

  1. bind bind用于配置允许连接Redis的网络地址 默认配置为127.0.0.1，只能在Redis本机访问，不能远程访问 如果将bind删除或直接注释，则Redis会接受所有可用网络的连接请求
  2. protected-mode 保护模式，默认开启。 关闭保护模式后，外部可用网络能够直接访问Redis 开启保护模式后，配置bind或使用密码后才能访问Redis
  3. port Redis监听的端口，默认是6379
  4. timeout 当客户端闲置一定时间后，关闭该连接。如果值为0，则表示禁止该功能。默认禁止

# 二、通用

  1. daemonize 是否以守护进程的方式启动Redis，默认为no 设置为yes后，Redis将以守护进行（后台运行）的方式运行
  2. pidfile 进程Id文件，默认为“/var/run/redis_6379.pid” 当Redis以守护进程模式启动时，会创建该文件，关闭Redis，会删除该文件 如果不是以守护进程启动Redis，则不会创建或删除该文件。
  3. loglevel 日志级别，默认为notice。可选值有：debug，verbose，notice，warning
  4. logfile 日志文件 如果配置为空字符串，则是标准输出 如果配置为空字符串，且是以守护进程模式启动Redis，则会将日志输出到/dev/null
  5. databases 设置数据库个数，默认是16个 默认使用的数据库是第0个，可以使用select dbid来切换数据库
  6. save 持久化设置，默认情况下，有三种情况会将数据写入磁盘（900秒，1次变更；300秒，10次变更；60秒，10000次变更）
  7. stop-writes-on-bgsave-error 如果持久化方案是rdb，在最后一次保存失败后，Redis将停止接受写操作，Redis保存进程重新启动工作后，Redis将自动允许写操作
  8. dbfilename RBD持久化方案的文件名称，默认是dump.rdb
  9. dir 指定备份的文件路径，备份文件名称就是dbfilename对应的名称

# 三、主从复制

  1. replicaof 主从同步配置，在从节点中配置主节点的地址
  2. masterauth 主节点的密码
  3. replica-serve-stale-data 当一个从节点与主节点失去连接的时候，或者同步正在进行中，slave的行为可能有两种：
    1. 如果replica-serve-stale-data设置为yes，slave会继续响应客户端请求，可能是正常数据，也可能是还没获得值得空数据
    2. 如果replica-serve-stale-data设置为no，slave会向客户端回复“正在从master同步数据”来处理各种请求
  4. replica-read-only 从节点是否只读
  5. repl-diskless-sync 数据同步策略 当一个新的slave加入后，需要进行一次全量的数据同步。同步策略主要有：
    1. 基于硬盘（disk-backed）：master创建一个新进程dump RDB，然后由父进程传输给slave。在dump RDB完成后，可以同时服务多个slave
    2. 基于socket（diskless）：master创建一个新进程直接dump RDB到slave的socket，不进过主进程，不进过磁盘。基于socket，只能单个进行 默认情况下，使用的是基于磁盘；如果网络较快，也可以使用基于socket
  6. repl-diskless-sync-delay 如果使用非磁盘同步，可以配置同步延迟时间，以等待master产生子进程通过socket传输RDB数据给slave。默认值为5秒，设置为0，则每次传输无延迟
  7. repl-ping-replica-period slave根据指定的时间间隔向master发送ping请求
  8. slave-priority 设置slave的优先级，默认为100。在master不能工作的时候，sentinel会从可用的slave中选择一个节点作为master，优先级数值小的slave会被优先考虑作为master 如果一个slave的优先级为0，则不会被提升为master

# 四、安全

  1. requirepass 设置访问密码
  2. rename-command 对命令重命名，可以将一个危险的命令重命名为其他名称。如果将一个命令重命名为空字符串，则是禁用此命令

# 五、连接限制

  1. maxclients 设置最大的连接数

# 六、内存管理

  1. maxmemory 设置最大内存
  2. maxmemory-policy 最大内存策略。当内存使用达到最大值时，所采取的策略，默认是noeviction。
    • volatile-lru：根据LUR算法，删除过期Key
    • allkeys-lru：根据LUR算法，删除任何key
    • volatile-random：随机删除过期key
    • allkeys-random：随机删除任意key
    • volatile-ttl：移除即将过期的key
    • noeviction：不删除任何key，向客户端返回一个错误（默认）

# 七、AppendOnly

  1. appendonly 是否开启appendonly，一种数据持久化方案
  2. appendfilename 持久化文件名称，默认文件名为："appendonly.aof"
  3. appendfsync 持久化策略，有三个选择项：
    • always：及时持久化
    • everysec：每秒写入一次
    • no：不会立刻刷，只有在操作系统需要刷的时候再刷。

# 八、集群

  1. cluster-enabled

    是否开启集群，只有开启后，当前节点才能作为集群的一部分

  2. cluster-config-file nodes-6379.conf

    集群配置文件

  3. cluster-node-timeout

    设置集群节点超时时间，超时的节点将被视为不可用状态

  4. cluster-slave-validity-factor

    如果数据太旧，集群中的不可用master的slave节点会避免成为备用master节点。如果slave和master失联时间超过（nodel-timeout * slave-validity-factor）+ repl-ping-period，则不会被提升为master节点。

    如果 slave-validity-factor的值比较大，则允许包含过旧数据的slave成为master，如果比较小，则可能会阻止集群选举出master节点。

  5. cluster-migration-barrier

    该属性可以保证集群中不会出现裸奔的主节点（没有从节点的主节点），当某个主节点的从节点宕机后，集群会从其他富余的主节点上分配一个从节点过来，确保每个主节点都有至少一个从节点，不至于因为主节点宕机而没有从节点来替换。该配置的默认值是1
