# Spring循环依赖

面试题：

  1. Spring中的三级缓存是什么，三个Map有什么异同
  2. 什么事循环依赖，一般说的Spring容器是什么
  3. 如何检测是否存在循环依赖，实际开发中遇到多循环依赖吗，循环依赖的异常是什么
  4. 多例的情况下，循环依赖问题为什么无法解决
  5. 为什么一定要用三级缓存，使用二级或者一级可否

什么是循环依赖：

多个bean之间相互依赖，形成了一个闭环。

通常来说，如果问Spring容器内部如何解决循环依赖，一定是指默认的单例Bean中，属性相互引用的场景

在一个Spring项目中，A引用B，B引用C，C引用A

package com.it.cycle;

import org.springframework.beans.factory.annotation.Autowired;
import org.springframework.stereotype.Component;

@Component
public class A {

    @Autowired
    private B b;
}

package com.it.cycle;

import org.springframework.beans.factory.annotation.Autowired;
import org.springframework.stereotype.Component;

@Component
public class B {

    @Autowired
    private C c;
}

package com.it.cycle;

import org.springframework.beans.factory.annotation.Autowired;
import org.springframework.stereotype.Component;

@Component
public class C {

    @Autowired
    private A a;
}

两种注入方式（构造注入与set注入）对循环依赖的影响：

官网解释

Circular dependencies

If you use predominantly constructor injection, it is possible to create an unresolvable circular dependency scenario.

For example: Class A requires an instance of class B through constructor injection, and class B requires an instance of class A through constructor injection. If you configure beans for classes A and B to be injected into each other, the Spring IoC container detects this circular reference at runtime, and throws a BeanCurrentlyInCreationException.

One possible solution is to edit the source code of some classes to be configured by setters rather than constructors. Alternatively, avoid constructor injection and use setter injection only. In other words, although it is not recommended, you can configure circular dependencies with setter injection.

Unlike the typical case (with no circular dependencies), a circular dependency between bean A and bean B forces one of the beans to be injected into the other prior to being fully initialized itself (a classic chicken-and-egg scenario).

如果在循环依赖的时候，使用setter注入，并且Bean是单例的，就能得到解决。

如果使用构造器注入，则无法解决循环依赖

默认的单例模式（Singleton），是支持循环依赖的，不会报错

原型模式（Prototype）不支持循环依赖，会报错

循环依赖的报错是：BeanCurrentlyInCreationException

在Spring内部，通过三级缓存来解决循环依赖（DefaultSingletonBeanRegistry）

/** Cache of singleton objects: bean name to bean instance. */
// 一级缓存（成品），也叫单例池，存放已经经历了完整生命周期的Bean对象 
private final Map<String, Object> singletonObjects = new ConcurrentHashMap<>(256);

/** Cache of early singleton objects: bean name to bean instance. */
// 二级缓存（半成品），存放早期暴露出来的Bean对象，Bean的生命周期还未结束，属性还未填充完毕
private final Map<String, Object> earlySingletonObjects = new ConcurrentHashMap<>(16);

/** Cache of singleton factories: bean name to ObjectFactory. */
// 三级缓存，存放可以生成Bean的工厂
private final Map<String, ObjectFactory<?>> singletonFactories = new HashMap<>(16);

只有单例Bean会通过三级缓存提前暴露出来解决循环依赖的问题，而非单例Bean，每次从容器中获取都是新创建的一个对象，所以非单例Bean是没有缓存的，不会将其放入三级缓存中。

Bean的创建过程（A、B相互依赖）：

  1. 在实例化A的过程中，需要B，此时A将自己放入第三级缓存中，去实例化B
  2. 在实例化B的过程中，需要A，于是B先查询第一级缓存，发现第一级缓存中没有A；再查第二级缓存，第二级缓存中也没有A；再查询第三级缓存，在第三级缓存中找到了A，然后把A存入二级缓存中，并删除第三级缓存中的A
  3. B实例化并初始化完成后，将自己存放在第一级缓存中（此时B里面的A状态是创建中），然后回来接着创建A，此时B已经创建完成，直接从一级缓存中获取B，然后A创建完成，并将A自己放入第一级缓存中。
